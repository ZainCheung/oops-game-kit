# 多Agent协作式游戏需求生成系统——技术方案

## 1. 概述

本方案旨在构建一套 **多Agent分层协作 + 结构化知识图谱 + 自动化验证** 的内部开发工具链，使策划人员在不接触任何技术术语的前提下，描述游戏功能需求，系统即可自动生成符合项目规范的可运行代码，并形成可度量、可维护的闭环。

核心思路：让Agent各司其职，让知识分层索引，让验证贯穿始终。

## 2. 系统架构总览

```text
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  策划输入    │────▶│  Agent A     │────▶│  人工确认    │
│  (自由文本)  │     │  业务理解    │     │  (需求描述)  │
└─────────────┘     └──────┬──────┘     └─────────────┘
                           │ 结构化需求包
                           ▼
                    ┌─────────────┐
                    │  Agent B     │
                    │  技术翻译    │
                    └──────┬──────┘
                           │ API调用蓝图
                           ▼
                    ┌─────────────┐     ┌─────────────┐
                    │  Agent C     │────▶│  自动化测试  │
                    │  代码生成    │     │  + 人工审核  │
                    └─────────────┘     └─────────────┘
                           │
                           ▼
                    最终代码/分支
```

所有Agent均挂载分层知识库与引用追溯模块，外部通过统一触发入口（CLI / IDE插件）调用。

## 3. 分层知识库设计

### 3.1 三层知识结构

| 层级 | 名称 | 内容 | 载体 |
|------|------|------|------|
| L1 | 策划业务知识库 | 功能黑盒描述、触发条件、业务规则、表现效果、异常处理 | Markdown + YAML头 |
| L2 | API对照知识库 | 业务动作/名词 → 稳定业务API映射、组合操作模板 | JSON / 结构化DB表 |
| L3 | 技术规范知识库 | 项目编码规范、架构约束、公共组件用法、错误处理模板 | Markdown + 规则脚本 |

### 3.2 知识关系图谱（程序索引）

三层知识之间通过 `knowledge_relations` 关系表串联，构成有向图。例如：

```yaml
- source: { type: L1, id: "guild-war-settlement" }
  relation: "maps_to_api"
  target: { type: L2, id: "send-guild-reward" }

- source: { type: L2, id: "send-guild-reward" }
  relation: "depends_on_spec"
  target: { type: L3, id: "mail-system-spec" }
```

关系表维护方式：

- 在L1/L2文档的YAML头部声明关联ID，由CI程序解析入库。
- 支持在后台管理界面手工维护。
- 每次知识更新后自动校验关联有效性（目标ID是否存在）。

## 4. 知识检索机制

### 4.1 检索策略

**程序做粗筛，大模型做精筛。** 避免让大模型盲目检索海量知识。

- **策划术语库**：采用 embedding 向量相似度 + 业务同义词映射，召回 Top-K 个标准术语。
- **API对照库**：结构化精确匹配，按 业务动作+名词 查表。
- **技术规范库**：全局常驻System Prompt + 按场景关键词（如"UI"、"网络"）触发动态加载。

### 4.2 关联知识扩展

检索到L1的某个术语后，通过知识关系表一次性拉出它关联的L2 API和L3规范，打包成 `knowledge_bundle` 返回给Agent。

检索服务提供两个核心API：

- `GET /search/L1?q=需求文本` → 返回 L1 术语列表
- `GET /bundle?term_id=xxx` → 返回该术语相关的 L1+L2+L3 知识包

## 5. 多Agent流水线设计

### 5.1 Agent A：业务理解与结构化

**职责**：将策划自由文本转换为标准业务需求描述包，附引用依据。

- **输入**：策划原始需求文本
- **携带知识**：L1知识库（通过检索服务按需获取）

**执行逻辑**：

1. 调用 `/search/L1` 获取候选术语与L1知识包。
2. 多轮对话或单次推理，产出结构化JSON：

```json
{
  "biz_module": "guild-war-settlement",
  "trigger": "on_guild_war_end",
  "preconditions": ["player_joined_guild"],
  "steps": [
    {"seq": 1, "action": "compare_rank", "params": "..."},
    {"seq": 2, "action": "发放道具", "params": {"item": "神之剑", "count": 1}}
  ],
  "data_changes": ["guild_funds +100"],
  "exceptions": [{"when": "背包满", "then": "发送特殊邮件"}],
  "presentation": "全服广播 + 飘字",
  "references": [
    {"type": "L1", "id": "guild-war-settlement", "excerpt": "..."},
    {"type": "L1", "id": "backpack-full-mail", "excerpt": "..."}
  ]
}
```

- **输出**：结构化业务需求包（供人工确认，也可直接传递给Agent B）。

### 5.2 Agent B：技术翻译与API映射

**职责**：将业务需求包翻译为API调用蓝图（伪代码级别），附映射依据。

- **输入**：Agent A输出的业务需求包
- **携带知识**：L2知识库（通过 bundle 接口获取相关API映射）

**执行逻辑**：

1. 解析需求包中的业务动作列表。
2. 对每个动作，精确查询L2库，获取对应的高层API模板。
3. 生成技术蓝图：

```json
{
  "steps": [
    {
      "seq": 1,
      "action": "调用 ItemManager.GrantItem(playerId, itemId, count)",
      "mapping_ref": "L2:action=发放道具 → ItemManager.GrantItem",
      "pre_check": "Inventory.CheckSlot()"
    },
    {
      "seq": 2,
      "condition": "slot_empty < 1",
      "action": "调用 MailManager.SendSpecial(playerId, templateId)",
      "mapping_ref": "L2:异常处理=背包满转邮件"
    }
  ]
}
```

- **输出**：API调用蓝图（供技术人员快速审查）。

### 5.3 Agent C：代码生成与规范注入

**职责**：根据蓝图生成目标语言代码，严格遵循技术规范。

- **输入**：Agent B输出的技术蓝图
- **携带知识**：L3技术规范（全局常驻 + 场景触发）

**执行逻辑**：

1. 将蓝图步骤展开为具体代码。
2. 强制应用规范：统一日志工具、异常处理模板、命名规范等。
3. 为关键约束处添加注释说明，如 `// 遵循UI规范: 多分辨率锚点适配`

- **输出**：完整代码文件，附带规范引用注释块。

## 6. 人工确认闸门

系统在以下节点设置人工确认，防止错误扩散：

- **闸门1**（Agent A输出后）：策划确认业务需求包，检查引用的L1知识是否正确，可高亮纠错。
- **闸门2**（Agent B输出后）：技术负责人审查API蓝图，检查映射是否张冠李戴，支持低置信度条目强制复核。

确认界面需展示每个Agent的 **知识引用清单**，并提供"纠错标记"功能，反馈直接写入知识库。

## 7. 测试与验证框架

### 7.1 知识引用准确率测试

- **黄金数据集**：针对L1、L2分别维护N组（输入 → 期望引用ID列表）
- **测试方式**：CI中运行pytest脚本，调用Agent的检索部分（不生成完整代码），比对实际引用ID与期望ID，计算准确率。
- **阈值告警**：准确率低于95%阻止部署新知识库。

### 7.2 代码规范硬性检查

- Lint工具（如ESLint、SonarQube）检查通用规范。
- 自定义规范脚本：例如检测是否使用 `GameLogger` 而非 `print`，是否包含网络超时参数。
- 在Agent C生成代码后自动触发，不通过则驳回。

### 7.3 业务逻辑动态验证

- **场景构造器**：基于需求自动生成最小化游戏测试环境（如特定玩家状态、背包情况）。
- **执行与翻译**：在测试环境运行生成代码，捕获日志并翻译为业务断言（"背包满时收到了特殊邮件"）。
- 封装为 `biz-test-skill`，作为流水线最后一步，结果反馈给策划/技术。

## 8. 上下文管理方案

为避免单Agent上下文爆炸，采取以下策略：

- **知识隔离**：每个Agent只加载本层知识库的检索结果，不携带全量知识。
- **分层加载**：摘要→骨干→细节，按需深入。
- **对话压缩**：多轮交互时，将历史对话压缩为结构化摘要，释放空间。
- **中间产物紧凑**：Agent A输出几百字的JSON，Agent B输出蓝图，不再保留原始对话。

## 9. 与现有工具链的集成

| 现有工具 | 集成方式 |
|----------|----------|
| Cursor / Trae | 自定义Command `/new-feature`，触发流水线，结果在终端或面板展示 |
| Cloud Code | 通过API调用云端Agent编排服务，支持"生成-测试-部署"一气呵成 |
| 自研Agent Skill | 将知识检索、API映射、规范注入分别封装为可复用Skill，供流水线调用 |
| CI/CD (GitHub Actions) | 自动化知识库索引更新、基准测试、Lint检查 |

## 10. 工作流概览（技术视角）

1. **知识更新**：在Git仓库修改Markdown/YAML → CI解析关系入库 → 重建索引 → 自动跑基准测试。
2. **策划需求**：在Cursor输入 `/new-feature 【需求文本】` → 触发编排引擎。
3. **A输出**：策划确认 → 若不通过则标错，重试。
4. **B输出**：技术审查 → 若不通过则标错，修正映射。
5. **C生成**：代码入分支 → 自动Lint + 业务测试。
6. **监控**：记录每次生成的引用ID和最终测试通过率，生成质量日报。

## 11. 风险与应对

| 风险 | 应对措施 |
|------|----------|
| 知识关系缺失导致引用断层 | 提供管理界面，允许快速补录关系；上线初期允许Agent请求人工补充 |
| 大模型对业务逻辑理解偏差 | 多轮确认+蓝图可视化审查，将偏差拦截在代码生成前 |
| 生成代码性能问题 | 技术规范中包含性能基线要求，并作为Lint规则检查 |
| 知识库过度膨胀 | 定期归档冷门模块，摘要化长尾知识，保持检索质量 |

## 12. 结语

本方案通过 **多Agent分工、知识图谱索引、程序化粗筛+模型精筛、全链路验证** 四大支柱，解决了复杂游戏业务中"上下文不足、生成不准、知识库难维护"的核心痛点。实施时可按阶段落地：先建L1知识库与Agent A，再逐步串联B/C与测试框架，最终形成高质量、可演进的策划驱动开发闭环。
