# 多Agent协作式游戏需求生成系统——模拟面试问答

## 一、系统架构设计

### Q1：为什么选择多Agent流水线而不是单Agent端到端生成？单Agent + 长上下文窗口现在也能处理很多任务了，你的方案是否过度设计？

**答：**

选择多Agent的核心原因有三点，而非过度设计：

1. **职责隔离降低幻觉风险**：单Agent同时承担"理解业务→映射API→生成代码"三件事，上下文中混杂策划术语、API签名、编码规范，模型极易混淆层级——比如把策划的"发放道具"直接翻译成不存在的API。多Agent各司其职，每个Agent的Prompt和知识域收窄，输出更可控。

2. **可验证性**：单Agent端到端输出最终代码，中间过程是黑盒，出错后无法定位是"理解错了"还是"映射错了"还是"代码写错了"。多Agent流水线每一步产出结构化中间物（需求包→蓝图→代码），可在对应闸门精准拦截和归因。

3. **知识库分层检索效率**：单Agent需要一次性加载L1+L2+L3全量知识，即使长上下文窗口能装下，检索精度也会下降（注意力稀释）。多Agent按需加载本层知识，检索空间小、精度高。

**权衡**：我承认多Agent增加了编排复杂度和通信开销。但游戏业务中"生成不准"的代价远高于"编排复杂"的代价——一行错误的游戏逻辑代码可能导致线上事故，而多一次JSON传递只是毫秒级延迟。

---

### Q2：Agent B发现Agent A的输出有歧义，需要回溯，流水线如何处理？

**答：**

设计两种回溯策略：

1. **轻量级回溯——B自行消歧**：如果歧义范围有限（比如某个action有两种理解），B携带歧义标记向A发起"澄清请求"，A仅针对歧义点补充推理，不需要重跑整个流程。类似函数调用中的局部重入。

2. **重量级回溯——退回A重跑**：如果B发现A的输出整体逻辑不自洽（比如前置条件和步骤矛盾），则将A的输出连同B的"异议说明"一起退回，A基于异议重新生成。此时A的Prompt中会附加"上次输出被退回的原因"，避免重复犯错。

实现上，编排引擎维护一个状态机：`A_RUNNING → A_DONE → B_RUNNING → B_NEEDS_CLARIFY → A_CLARIFYING → B_RUNNING → B_DONE → ...`，每个状态转换都有超时兜底，超时则升级为人工介入。

---

### Q3：三个Agent是串行执行还是有并行可能？

**答：**

默认串行，因为存在严格的数据依赖（B依赖A的输出，C依赖B的输出）。但在以下场景可并行：

1. **需求拆分并行**：如果策划输入涉及多个独立模块（如"公会战结算 + 跨服排行榜"），编排引擎先做需求拆分，然后A1处理公会战、A2处理排行榜，并行运行后合并输出给B。

2. **知识预加载并行**：A运行时，编排引擎可以预判B可能需要的L2知识，提前发起检索请求，与A的执行并行，减少B的等待时间。

3. **C的代码生成与Lint并行**：C生成代码后，Lint检查和业务测试可以并行执行。

实际落地中，第一种并行（需求拆分）是最有价值的，但需要编排引擎具备需求分解能力，这是MVP之后优化的方向。

## 二、知识库与检索

### Q4：embedding召回的Top-K是多少？K值怎么定的？召回率和精确率怎么权衡？

**答：**

- **K值设定**：默认K=10，但会根据查询复杂度动态调整。简单查询（如"背包满"）K=5即可，复杂查询（如"公会战结算异常处理"）K=15。判断复杂度的依据是查询文本的实体数量和修饰词数量。

- **K值调优方法**：基于黄金数据集做离线评估，画出Recall@K曲线，选择收益拐点。实测中K从5到10召回率提升显著（约15%），10到20提升趋缓（约3%），所以默认10是性价比最高的。

- **精筛策略**：Top-K召回后，不是全部塞给大模型，而是用大模型做一次re-rank，选出Top-3最相关的条目。这样既保证了召回率（粗筛不漏），又控制了输入长度（精筛不冗）。

---

### Q5：三种不同检索策略，如何保证跨层检索的一致性？

**答：**

跨层一致性由知识关系图谱保证，而非检索策略本身。具体流程：

1. L1向量检索返回术语ID列表。
2. 对每个术语ID，通过关系表查询关联的L2 API ID和L3规范ID。
3. 用ID去L2/L3做精确查找，如果找不到，说明关系断裂。

**L1召回但L2找不到映射的情况**：这是预期内的正常场景，处理方式：
- 标记为"映射缺失"，在Agent B的输出中显式标注 `mapping_status: "missing"`。
- 触发知识库维护提醒，通知技术负责人补充L2条目。
- Agent B降级处理：基于L1的业务描述自行推断可能的API调用，但标注低置信度，在闸门2强制人工复核。

这比"强行匹配一个错误API"安全得多。

---

### Q6：L1术语一对多或多对一映射到L2，Agent B如何消歧？

**答：**

**一对多（一个业务动作对应多个API）**：
- 这通常是正常的组合操作，不是歧义。比如"发放公会奖励"可能映射到 `GrantItem` + `AddGuildFunds` + `SendMail` 三个API，Agent B应全部调用。
- L2知识库中用"组合操作模板"预定义这类映射，而非让Agent自行拆解。

**多对一（多个业务动作对应同一个API）**：
- 这是真正的消歧场景。比如"发放道具"和"发放奖励"都映射到 `GrantItem`，但参数不同。
- 消歧依据来自Agent A输出的结构化需求包中的 `params` 字段——A已经明确了具体参数（item、count等），B根据参数选择正确的API调用方式。
- 如果参数仍不足以消歧，B标记为"需确认"，在闸门2由技术负责人裁定。

---

### Q7：删除L2条目时，有10个L1条目引用它，怎么处理？

**答：**

采用"软删除 + 级联告警"策略：

1. **软删除**：L2条目标记为 `deprecated` 而非物理删除，保留7天过渡期。过渡期内检索仍可命中，但返回结果会附带弃用警告。
2. **级联告警**：删除操作触发关联检查，列出所有引用该L2条目的L1条目，生成告警清单推送给知识库维护者。
3. **强制阻断**：如果引用该L2的L1条目超过阈值（如5个），阻止删除操作，要求维护者先处理所有引用关系（迁移到替代API或更新L1描述）。
4. **过渡期结束**：7天后若引用已清零，自动物理删除；若仍有引用，升级为阻断并通知技术负责人。

这样既保证了知识库的演进自由度，又防止了"删了一个API，10个功能生成全挂"的灾难。

## 三、Agent 设计细节

### Q8：references字段是Agent自己填的还是检索服务返回的？如果Agent幻觉了一个不存在的L1 ID，下游怎么发现？

**答：**

**来源**：references由检索服务返回，Agent A只做选择和引用，不自行编造ID。具体流程：
1. Agent A调用 `/search/L1` 获取候选术语列表（含ID）。
2. Agent A从中选择与需求相关的术语，将ID和摘要填入references。
3. Agent A的Prompt中明确约束："只能引用检索服务返回的ID，禁止自行编造。"

**幻觉检测**：即使有约束，仍需防御性校验：
- 编排引擎在A输出后，自动校验references中每个ID是否存在于知识库中。
- 不存在的ID标记为 `invalid_reference`，触发告警并要求A重新生成。
- 这是轻量级的ID存在性检查，成本极低但能有效拦截幻觉。

**更深层的问题**——ID存在但引用错误（比如引用了不相关的L1条目）：这需要人工在闸门1确认，系统提供"引用高亮"功能，让策划快速判断引用是否合理。

---

### Q9：某个业务动作在L2库里没有对应API，系统怎么处理？

**答：**

分三级处理：

1. **降级生成 + 强制人工复核**：Agent B基于L1的业务描述和L3的编码规范，自行推断可能的实现方式，生成蓝图但标注 `confidence: low` 和 `mapping_status: "inferred"`。在闸门2强制要求技术负责人审查。

2. **触发知识库补充流程**：自动生成一条"待补充API映射"工单，包含业务动作描述和建议的API签名，推送给技术负责人。技术负责人可以选择：
   - 补充L2条目（长期方案）
   - 手动编写该段代码（短期方案）

3. **记忆与学习**：如果技术负责人在审查时修正了映射，该修正会被记录为"临时映射"，积累到一定频次后提示维护者正式入库。

核心原则：**宁可多一次人工确认，也不让系统在不确定的情况下静默生成错误代码。**

---

### Q10：如何保证Agent C真的遵循了L3规范，而不是"看起来遵循"？

**答：**

分三层验证：

1. **静态检查（Lint层）**：检测表面规范——命名规范、日志工具使用、异常处理模板等。这层只能验证"形式合规"。

2. **结构化约束注入**：不是让Agent C"记住规范然后遵守"，而是把规范转化为代码骨架（scaffold），Agent C只需填充业务逻辑。比如L3规范要求"所有网络请求必须包含超时参数和重试逻辑"，Agent C的Prompt中会附带网络请求的模板代码，Agent C在模板基础上修改，而非从零编写。这大幅降低了"忘记遵守"的概率。

3. **行为测试（业务逻辑层）**：生成的代码在测试环境中运行，验证其行为是否符合规范要求。比如规范要求"网络超时后重试3次"，测试用例模拟超时场景，检查重试次数。

**无法完全验证的规范**（如"代码可读性"）：这部分依赖代码审查，但通过前两层过滤，审查者只需关注逻辑正确性，不需要逐行检查格式规范。

---

### Q11：三个Agent用同一个大模型还是不同模型？

**答：**

**推荐不同模型，但MVP阶段用同一模型。**

选型逻辑：

| Agent | 核心能力要求 | 推荐模型类型 |
|-------|-------------|-------------|
| Agent A | 语义理解、业务推理 | 擅长理解的通用大模型（如GPT-4o、Claude） |
| Agent B | 结构化映射、逻辑推理 | 同上，但对代码能力要求不高 |
| Agent C | 代码生成、规范遵循 | 擅长代码的模型（如DeepSeek-Coder、GPT-4） |

**编排方式**：编排引擎为每个Agent配置模型ID，通过统一API接口调用，上层逻辑不感知底层模型差异。

**MVP策略**：先用一个综合能力强的模型（如GPT-4o）跑通全流程，验证架构可行性。后续根据各Agent的实际瓶颈针对性替换——如果A的理解能力够但C的代码质量差，只换C的模型。

## 四、人工确认闸门

### Q12：两个人工闸门拖慢流程，怎么平衡质量和效率？

**答：**

**渐进式自动化策略**：

1. **置信度分级**：每个Agent输出附带置信度评分。高置信度（>0.9）的条目可跳过闸门，低置信度（<0.7）的强制人工确认，中等置信度的抽样确认。

2. **批量确认**：不是每条需求都逐个确认，而是将一批需求的确认项汇总成一个审查清单，技术负责人一次性处理。类似Code Review的批量模式。

3. **异步确认**：闸门不阻塞流水线，而是"生成后审查"。代码先进入功能分支，人工确认通过后才合入开发分支。这样生成流程不等待，但合并流程有保障。

4. **信任积累**：对于历史确认通过率高的模块（如UI模块），逐步降低确认频率；对于高频出错的模块（如战斗逻辑），保持高确认频率。

**目标**：初期100%人工确认（建立信任基线），稳定后降至30%抽样确认，最终目标10%以下。

---

### Q13：策划纠错数据如何回流到知识库？

**答：**

两条路径，根据纠错类型分流：

1. **L1文档错误（策划可自行修复）**：纠错标记直接生成L1文档的修改建议（diff格式），策划在管理界面确认后自动更新L1文档，触发CI重建索引。

2. **L1术语缺失（需要新增条目）**：纠错标记转为"待补充术语"工单，由知识库维护者审核后新增L1条目，并建立与L2/L3的关联关系。

**不直接微调模型**的原因：
- 微调需要大量高质量数据，单次纠错数据量不够。
- 微调后的模型行为不可预测，可能影响其他场景。
- 纠错数据的价值在于"修正知识源"而非"修正模型"，修正知识源后所有Agent都受益。

长期积累的纠错数据可以作为微调数据集的候选，但这是二期优化，不是MVP重点。

---

### Q14：闸门2发现API映射错误，修改L2知识库还是修改Agent B的Prompt？

**答：**

**判断标准：这是"知识问题"还是"推理问题"？**

- **知识问题**（L2库中映射本身错误或缺失）→ 修改L2知识库。标志：Agent B正确引用了L2条目，但L2条目指向了错误的API。
- **推理问题**（L2映射正确，但Agent B选错了映射）→ 优化Agent B的Prompt或检索策略。标志：L2库中有正确映射，但Agent B没有选中。

**实际操作**：闸门2的审查界面会展示"Agent B引用的L2条目"和"可用的L2候选条目"，审查者可以直观判断是映射缺失还是选择错误，分别走不同修复路径。

**经验法则**：80%的映射错误是知识问题（L2缺失或过期），20%是推理问题。所以优先投入知识库维护，而非过度优化Prompt。

## 五、测试与验证

### Q15：95%准确率阈值是怎么来的？不同模块的难度差异怎么处理？

**答：**

95%是初始保守阈值，来源：
- 行业经验：搜索/推荐系统的召回准确率通常要求90%以上，知识检索场景要求更高（因为下游是代码生成，错误会被放大）。
- 实测校准：在黄金数据集上跑基线，当前系统的准确率约为92%，设定95%作为目标阈值，留出3%的改进空间。

**不同模块的差异化处理**：
- 按模块设定不同阈值：核心战斗模块98%、经济系统95%、UI模块90%。
- 阈值配置存储在知识库的模块元数据中，CI测试时按模块分别计算和判断。
- 新模块初期不设阈值（数据不足），积累到N条（如50条）黄金数据后再设定。

---

### Q16：场景构造器如何保证覆盖关键边界条件？

**答：**

**坦诚说，这是整套系统中最难的部分，无法做到100%覆盖，但可以通过以下策略逼近：**

1. **基于L1知识的边界枚举**：L1文档中的 `exceptions` 字段显式列出了异常场景（如"背包满"），场景构造器优先覆盖这些显式边界。

2. **基于L3规范的隐式边界**：L3规范中定义了技术边界（如"网络超时3秒"），构造器自动生成对应的测试场景。

3. **变异测试**：对正常场景的参数做边界变异（如count=0、count=MAX_INT、playerId=null），自动生成边界测试用例。

4. **历史Bug回归**：将历史Bug报告转化为测试场景，加入场景库。这是最实用的覆盖来源。

5. **覆盖率度量**：统计"已覆盖的L1异常条目 / L1异常条目总数"作为覆盖率指标，低于80%则告警。

**无法覆盖的场景**：并发竞态、时序依赖等复杂交互场景，仍需人工编写集成测试。场景构造器的定位是"覆盖80%的常见边界"，而非替代所有测试。

### Q17：上线后出了bug，如何做根因分析？

**答：**

**全链路追溯机制**：

每次生成都会记录一条完整的"生成谱系"（lineage），包含：

```
生成ID: gen-20240613-001
├── Agent A 输出: {需求包, references: [L1:guild-war-settlement, L1:backpack-full-mail]}
├── Agent B 输出: {蓝图, mapping_refs: [L2:send-guild-reward, L2:mail-fallback]}
├── Agent C 输出: {代码文件, spec_refs: [L3:mail-system-spec]}
├── 测试结果: {Lint: pass, 业务测试: pass}
└── 人工确认: {闸门1: 策划张三确认, 闸门2: 技术李四确认}
```

Bug发生后的根因分析流程：

1. **定位Bug代码**：通过代码文件找到对应的生成ID。
2. **回溯谱系**：查看该代码由哪个蓝图生成，蓝图由哪个需求包翻译，需求包引用了哪些L1/L2/L3知识。
3. **分类归因**：
   - L1知识错误 → 策划维护L1
   - L2映射错误 → 技术维护L2
   - L3规范缺失 → 架构组补充L3
   - Agent推理错误 → 优化Prompt或模型
   - 测试未覆盖 → 补充测试场景
4. **修复与防复发**：修复Bug的同时，在对应层级补充知识或测试用例，防止同类问题再次生成。

## 六、上下文管理

### Q18：对话压缩的具体策略是什么？摘要丢失的关键信息怎么补偿？

**答：**

**压缩策略**：

1. **结构化摘要**：不是简单的文本摘要，而是提取为固定Schema的结构化数据。比如Agent A的多轮对话压缩为最终的JSON需求包，丢弃中间推理过程。

2. **关键决策保留**：压缩时保留"决策节点"——即Agent做出选择及其理由。比如"选择了'发放道具'而非'发放货币'，因为L1:reward-type-spec指定了道具类型"。

3. **增量对话**：多轮交互时，只传递新增内容+结构化摘要，不重复传递已有内容。

**摘要丢失的补偿**：

- 如果下游Agent需要上游的推理细节，可以通过"按需回溯"机制获取——编排引擎保存了完整的对话历史，下游Agent可以发起"上下文查询"请求，获取特定节点的原始对话。
- 但这应该是例外而非常态，常态下结构化中间产物应包含足够信息。

---

### Q19：Agent B需要Agent A推理过程中的上下文怎么办？

**答：**

**设计原则：中间产物应自包含，不依赖上游推理上下文。**

如果B确实需要A的推理信息，说明A的输出格式不够完善，应该优化A的输出Schema，而非让B回溯A的对话。

具体做法：
- 在A的输出Schema中增加 `reasoning_notes` 字段，记录关键推理依据（如"排除了X理解，因为Y原因"）。
- B的Prompt中明确说明"可参考reasoning_notes辅助理解"。

如果发现B频繁需要A的推理细节，这是A的输出设计有问题的信号，应该复盘并优化A的结构化输出格式，而非建立常态化的回溯通道。

## 七、工程落地

### Q20：从0到1落地，优先级怎么排？MVP是什么形态？

**答：**

**三阶段落地计划：**

**阶段1（MVP，核心闭环）**：
- 建L1知识库（只覆盖1-2个核心模块，如邮件系统和背包系统）
- 实现Agent A（业务理解+结构化输出）
- 实现Agent C（简化版，跳过B，A直接给C，API映射写在Prompt中）
- 1个人工闸门（A输出后确认）
- 基础Lint检查
- **MVP形态**：CLI命令 `/new-feature 需求文本` → A生成需求包 → 人工确认 → C生成代码 → Lint检查

**阶段2（完善流水线）**：
- 引入Agent B，建立L2知识库
- 增加闸门2
- 实现知识关系图谱和关联检索
- IDE插件集成

**阶段3（自动化验证）**：
- 实现场景构造器和业务逻辑测试
- 置信度分级和渐进式自动化
- 质量监控和日报

**砍掉的优先级**：场景构造器最复杂且非核心，MVP中完全可以用人工测试替代。

---

### Q21：IDE插件交互空间有限，策划确认和蓝图审查体验差，怎么解决？

**答：**

**分角色设计交互**：

- **策划**：不直接在IDE中操作。策划通过Web管理界面进行确认，界面提供富文本展示、高亮标注、一键纠错等功能。IDE只负责触发流程和展示结果摘要。

- **技术负责人**：在IDE中做快速审查（蓝图以diff形式展示），复杂审查跳转到Web界面。

**触发方式**：IDE中 `/new-feature` 触发后，系统生成一个审查链接，推送到飞书/钉钉，策划在Web端确认后，IDE自动继续流程。

这样IDE只做"触发+展示"，复杂交互交给Web端，各取所长。

---

### Q22：知识库维护成本谁来承担？怎么激励团队持续维护？

**答：**

**成本分配**：

| 知识层 | 维护者 | 日常成本 |
|--------|--------|----------|
| L1 | 策划 | 写需求时顺手维护，增量成本极低（L1本身就是需求文档的规范化版本） |
| L2 | 技术负责人 | 新增/修改API时同步更新L2，属于文档工作的一部分 |
| L3 | 架构组 | 编码规范变更时更新，频率低 |

**关键洞察**：L1知识库不是额外工作，而是策划需求文档的"结构化升级版"。策划本来就要写需求文档，L1只是要求按固定格式写，反而减少了后续的沟通成本。

**激励机制**：
- **正向**：维护好的模块，生成代码的准确率更高，策划提需求后等待时间更短——这是自驱动力。
- **负向**：未维护L1的模块，生成代码强制走全量人工确认流程，效率更低——这是倒逼机制。
- **度量**：每月公布各模块的"知识健康度"（覆盖率、更新频率、引用准确率），形成团队内的良性竞争。

## 八、开放性问题

### Q23：如果重新设计，会做哪些不同的选择？

**答：**

1. **更早引入Agent B的"预校验"机制**：当前方案中B只在A输出后工作，但如果在A运行时B就并行做"预映射检查"（检查A引用的L1术语是否都有L2映射），可以更早发现映射缺失，避免A输出后才发现无法翻译。

2. **知识库版本化**：当前方案中知识库更新后直接替换，没有版本管理。应该给知识库加版本号，每次生成记录使用的知识库版本，这样Bug回溯时可以还原当时的知识状态。

3. **更激进的MVP**：MVP阶段甚至可以跳过向量检索，直接用关键词匹配做L1检索，先把流水线跑通，再优化检索质量。过早优化检索是常见的陷阱。

---

### Q24：最大的技术风险是什么？3个月砍掉什么？

**答：**

**最大技术风险**：知识库的冷启动问题。系统效果高度依赖知识库的质量和覆盖度，但初期知识库几乎是空的，导致Agent检索不到足够的上下文，生成质量差，进而导致团队对系统失去信任，不愿投入维护知识库——形成恶性循环。

**应对**：从最成熟、文档最完善的模块开始（而不是最复杂的模块），确保首批体验就有明显效果，建立团队信心。

**3个月砍掉清单**：
- 场景构造器和业务逻辑动态验证（最复杂，用人工测试替代）
- 知识关系图谱的可视化管理界面（用YAML文件+CI校验替代）
- 置信度分级和渐进式自动化（初期100%人工确认）
- 多模型编排（初期用同一模型）
- 需求拆分并行（初期只处理单模块需求）

3个月的核心目标：**1个模块（邮件系统）从需求到代码的全流程跑通，且生成代码通过人工审查。**

### Q25：和LangGraph、CrewAI、AutoGen等开源方案相比，核心差异在哪？

**答：**

| 维度 | 通用Agent框架 | 本方案 |
|------|-------------|--------|
| 定位 | 通用Agent编排框架 | 游戏业务专用的需求→代码生成系统 |
| 知识管理 | 无内置方案，需自行接入向量数据库 | 三层分层知识库+关系图谱，与Agent深度耦合 |
| 验证机制 | 无内置方案 | 全链路验证（引用准确率+Lint+业务测试） |
| 人工介入 | 通常无结构化闸门 | 两个人工确认闸门，支持纠错回流 |
| 领域适配性 | 需大量Prompt工程适配游戏业务 | L1/L2/L3知识库本身就是领域适配层 |

**核心差异**：通用框架解决的是"怎么编排Agent"，本方案解决的是"怎么让Agent在游戏业务中生成正确的代码"。前者是基础设施，后者是领域方案。

**实际上可以结合使用**：用LangGraph做底层编排引擎，本方案的知识库、闸门、验证框架作为上层领域逻辑。不必重复造轮子，但领域知识层是通用框架无法替代的。
